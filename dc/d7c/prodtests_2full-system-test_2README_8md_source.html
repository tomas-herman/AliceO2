5 ## Full system test configuration and scripts
6 
7 The full system test workflow scripts consist of 3 shell scripts:
8 * `dpl-workflow.sh` : The main script that runs the dpl-workflow for the reconstruction.
9  It can read the input either internally, or receive it externally by one of the others.
10 * `raw-reader.sh` : Runs the `o2-raw-file-reader` to read the raw files as external input to `dpl-workflow.sh`.
11 * `datadistribution.sh` : Run the `StfBuilder` to read time frame files as external input to `dpl-workflow.sh`.
12 
13 One can either run the `dpl-workflow.sh` standalone (with `EXTINPUT=0`) or in parallel with one of the other scripts in separate shells (with `EXTINPUT=1`)
14 
15 In addition, there is the shared `setenv.sh` script which sets default configuration options, and there is the additional benchmark script:
16 * `start_tmux.sh` : This starts the full test in the configuration for the EPN with 2 NUMA domains, 512 GB RAM, 8 GPUs.
17  It will run tmux with 3 sessions, running twice the `dpl-workflow.sh` and once one of the external input scripts (selected via `dd` and `rr` command line option).
18  * Please note that `start_tmux.sh` overrides several of the environment options (see below) with the defaults for the EPN.
19  The only relevant options for `start_tmux.sh` should be `TFDELAY` and `GPUMEMSIZE`.
20  * Note also that while `dpl-workflow.sh` is a generic flexible script that can be used for actual operation, `start_tmux.sh` is a benchmark script to demonstrate how the full workflow is supposed to run on the EPN.
21  It is meant for standalone tests and not to really start the actual processing on the EPN.
22 
23 The `dpl-workflow.sh` can run both the synchronous and the asynchronous workflow, selected via the `SYNCMODE` option (see below), but note the following constraints.
24 * By default, it will run the full chain (EPN + FLP parts) such that it can operate as a full standalone benchmark processing simulated raw data.
25 * In order to run only the EPN part (skipping the steps that will run on the FLP), an `EPNONLY` option will be added later.
26 
27 All settings are configured via environment variables.
28 The default settings (if no env variable is exported) are defined in `setenv.sh` which is sourced by all other scripts.
29 (Please note that `start_tmux.sh` overrides a couple of options with EPN defaults).
30 The following options exist (some of the options are not used in all scripts, and might behave slightly differently as noted):
31 * `NTIMEFRAMES`: Number of time frames to process.
32  * `dpl-workflow.sh` without `EXTINPUT`: Will replay the timeframe `NTIMEFRAMES` time and then exit.
33  * `raw-reader.sh` : Will replay the timeframe `NTIMEFRAMES` time and `raw-reader.sh` will exit, the dpl-workflows will keep running.
34  * Ignored by `datadistribution.sh`, it will always run in an endless loop.
35 * `TFDELAY`: Delay in seconds between publishing time frames (1 / rate).
36 * `NGPUS`: Number of GPUs to use, data distributed round-robin.
37 * `GPUTYPE`: GPU Tracking backend to use, can be CPU / CUDA / HIP / OCL / OCL2.
38 * `SHMSIZE`: Size of the global shared memory segment.
39 * `DDSHMSIZE`: Size of shared memory unmanaged region for DataDistribution Input.
40 * `GPUMEMSIZE`: Size of allocated GPU memory (if GPUTYPE != CPU)
41 * `HOSTMEMSIZE`: Size of allocated host memory for GPU reconstruction (0 = default).
42  * For `GPUTYPE = CPU`: TPC Tracking scratch memory size. (Default 0 -> dynamic allocation.)
43  * Otherwise : Size of page-locked host memory for GPU processing. (Defauls 0 -> 1 GB.)
44 * `SAVECTF`: Save the CTF to a root file.
45 * `CREATECTFDICT`: Create CTF dictionary.
46  * 0: Read `ctf_dictionary.root` as input.
47  * 1: Create `ctf_dictionary.root`. Note that this was already done automatically if the raw data was simulated with `full_system_test.sh`.
48 * `SYNCMODE`: Run only reconstruction steps of the synchronous reconstruction.
49  * Note that there is no `ASYNCMODE` but instead the `CTFINPUT` option already enforces asynchronous processing.
50 * `NUMAGPUIDS`: NUMAID-aware GPU id selection. Needed for the full EPN configuration with 8 GPUs, 2 NUMA domains, 4 GPUs per domain.
51  In this configuration, 2 instances of `dpl-workflow.sh` must run in parallel.
52  To be used in combination with `NUMAID` to select the id per workflow.
53  `start_tmux.sh` will set up these variables automatically.
54 * `NUMAID`: SHM segment id to use for shipping data as well as set of GPUs to use (use `0` / `1` for 2 NUMA domains, 0 = GPUS `0` to `NGPUS - 1`, 1 = GPUS `NGPUS` to `2 * NGPUS - 1`)
55 * 0: Runs all reconstruction steps, of sync and of async reconstruction, using raw data input.
56 * 1: Runs only the steps of synchronous reconstruction, using raw data input.
57 * `EXTINPUT`: Receive input from raw FMQ channel instead of running o2-raw-file-reader.
58  * 0: `dpl-workflow.sh` can run as standalone benchmark, and will read the input itself.
59  * 1: To be used in combination with either `datadistribution.sh` or `raw-reader.sh` or with another DataDistribution instance.
60 * `CTFINPUT`: Read input from CTF ROOT file. This option is incompatible to EXTINPUT=1. The CTF ROOT file can be stored via SAVECTF=1.
61 * `NHBPERTF`: Time frame length (in HBF)
62 * `GLOBALDPLOPT`: Global DPL workflow options appended to o2-dpl-run.
63 * `EPNPIPELINES`: Set default EPN pipeline multiplicities.
64  Normally the workflow will start 1 dpl device per processor.
65  For some of the CPU parts, this is insufficient to keep step with the GPU processing rate, e.g. one ITS-TPC matcher on the CPU is slower than the TPC tracking on multiple GPUs.
66  This option adds some multiplicies for CPU processes using DPL's pipeline feature.
67  The settings were tuned for EPN processing with 8 GPUs.
68  It is auto-selected by `start-tmux.sh`.
69 * `SEVERITY`: Log verbosity (e.g. info or error)
70 * `SHMTHROW`: Throw exception when running out of SHM memory.
71  It is suggested to leave this enabled (default) on tests on the laptop to get an actual error when it runs out of memory.
72  This is disabled in `start_tmux.sh`, to avoid breaking the processing while there is a chance that another process might free memory and we can continue.
73 * `NORATELOG`: Disable FairMQ Rate Logging.
