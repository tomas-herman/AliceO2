5 # TrackExtrap.h(cxx)
6 
7 Contains all the tools to:
8 - extrapolate the track parameters (and covariances) from one z position to another, taking into account or not the
9 magnetic field.
10 - add the dispersion from Multiple Coulomb Scattering (MCS) to the covariance matrix when crossing a tracking chamber.
11 - extrapolate the track parameters (and covariances) through the front absorber, taking into account (or not) the
12 effects from MCS and energy loss.

14 # TrackFitter.h(cxx)
15 
16 Fit a track to the clusters attached to it, using the Kalman Filter.
17 
18 ## Short description of the algorithm
19 
20 The seeds for the track parameters and covariances are computed using the position and resolution of the 2 clusters in
21 the last two fired chambers. An estimate of the bending momentum is obtained by assuming that the track is coming from
22 the vertex (0,0,0) and crosses a constant magnetic field (both with sufficiently large uncertainties to do not bias the
23 final result).
24 The initial track parameters and covariances are then propagated to every clusters, up to the most upstream one, and are
25 updated at each step by "attaching" the cluster with the Kalman filter. The dispersion from MCS is added to the
26 covariances each time a chamber is crossed.
27 This algorithm gives the track parameters and covariances at the first cluster (the last attached one). Optionally, the
28 Smoother algorithm can be run to fit back the track and compute the track parameters and covariances at every clusters.
29 If the pointer to the parameters at one of the attached clusters is given, the fit is resumed starting from this point.

31 # TrackFinderOriginal.h(cxx)
32 
33 This is a reimplementation of the original tracking algorithm from AliRoot.
34 
35 ## Input / Output
36 
37 It takes as input a list of reconstructed clusters per chambers, the clusters' position being given in the global
38 coordinate system. It returns the list of reconstructed tracks, each of them containing (in the TrackParams) the
39 pointers to the associated clusters.
40 
41 ## Short description of the algorithm
42 
43 The original tracking algorithm is described in detail here:
44 https://edms.cern.ch/ui/file/1054937/1/ALICE-INT-2009-044.pdf
45 
46 The main difference with the original implementation is in the order the tracking of the initial candidates (made of
47 clusters in stations 4&5) is performed. In the original implementation, all candidates are propagated to station 3 to
48 search for compatible clusters, tracks are duplicated to consider all possibilities, then all candidates are propagated
49 to station 2, and so on. In some events with very high multiplicity, it results in a temporarily huge number of
50 candidates in memory at the same time (>10000) and a limitation is needed to abort the tracking in such case to do not
51 break the reconstruction. In the new implementation, each candidate is treated one by one. The current candidate is
52 propagated to station 3 to search for compatible clusters, duplicated to consider every posibilities, then each of them
53 is tracked one by one to station 2, then to station 1. This limits drastically the number of candidates in memory at the
54 same time, thus avoiding to abort the tracking for any event, while reconstructing exactly the same tracks in the end.

56 # TrackFinder.h(cxx)
57 
58 This is the implementation of the new tracking algorithm.
59 
60 ## Input / Output
61 
62 It takes as input a list of reconstructed clusters mapped per DE, the clusters' position being given in the global
63 coordinate system. It returns the list of reconstructed tracks, each of them containing (in the TrackParams) the
64 pointers to the associated clusters.
65 
66 ## Short description of the algorithm
67 
68 ### General ideas to improve the original algorithm:
69 - Consider the overlaps between DE on the same chamber during the tracking (the original algorithm considers only one
70 cluster per chamber during the tracking, duplicating the tracks with 2 clusters in overlapping DE to consider both), so
71 there is no need to complete the tracks to add the missing clusters afterward and to cleanup the duplicates. We assume
72 there is no overlap between the 2 half-planes of chambers 5 to 10.
73 - Do not try to recover tracks by removing one of the previously attached clusters in the previous station if no
74 compatible cluster is found in the current one. Assume that a good track will not be lost if several clusters are
75 attached in a station and some of them are bad (i.e. not belonging to that track).
76 - Further reduce the number of track duplication and memory consumption by pursuing the tracking of the current
77 candidate with the newly found compatible clusters down to station 1, or until the tracking fails, before testing other
78 clusters, so that we duplicate the original candidate only to store valid complete tracks.
79 - Speedup the cleanup of connected tracks in the end by reducing the number of imbricated loops.
80 - Speedup the track extrapolation in the magnetic field by reducing the number of steps to reach to desired z position.
81 
82 ### Algorithm:
83 - Build candidates on station 5 starting from pairs of clusters on chambers 9 and 10 and looking for compatible clusters
84 in overlap DE. Consider every combinations of different clusters and discard candidates whose parameters are outside of
85 acceptance limits within uncertainties.
86 - Propagate the candidates to station 4 and look for compatible clusters, taking into account overlaps. At least one
87 cluster is requested and the track is duplicated to consider every possibilities, eliminating the ones driving the track
88 parameters outside of acceptance limits within uncertainties.
89 - Repeat the two steps above starting from station 4 then going to station 5 to look for additional candidates with only
90 one chamber fired on station 5.
91 - Propagate each candidate found in the previous steps from chamber 6 to 1, looking for compatible clusters taking into
92 account the overlaps. This is a recursive procedure. For each cluster or couple of clusters found on one chamber, the
93 current parameters of the initial candidate are updated and the tracking continue to the next chamber, and so on and so
94 forth, until chamber 1 or until the tracking fails (at least one cluster per station is requested to continue). Only at
95 the end, when/if the track reaches the first station, it is duplicated and the clusters found in the process are
96 attached to the new track. Clusters that drive the track parameters outside of acceptance limits within uncertainties
97 are discarded.
98 - Improve the tracks: run the smoother to recompute the local chi2 at each cluster, remove the worst cluster if it does
99 not pass a stricter chi2 cut, refit the track and repeat the procedure until all clusters pass the cut or one of them
100 cannot be removed without violating the tracking conditions (by default, the track must contain at least 1 cluster per
101 station and both chambers fired on station 4 or 5), in which case the track is removed.
102 - Remove connected tracks in station 3, 4 and 5. If two tracks share at least one cluster in these stations, remove the
103 one with the smallest number of fired chambers or with the highest chi2/(ndf-1) in case of equality, assuming it is a
104 fake track.
105 
106 In all stations, the search for compatible clusters is done in a way to consider every possibilities, i.e. every
107 combinations of 1 to 4 clusters, while skipping the already tested combinations. This includes subsets of previously
108 found combinations, so that we always attach the maximum number of clusters to a candidate (e.g. if we can attach 4
109 clusters in one station because of the overlaps, we do not consider the possibility to attach only 3, 2, or 1 of them
110 even if the track would still be valid).
111 
112 A more detailed description of the various parts of the algorithm is given in the code itself.
113 
114 ### Available options:
115 - Find more track candidates, with only one chamber fired on station 4 and one on station 5, taking into account the
116 overlaps between DE and excluding those whose parameters are outside of acceptance limits within uncertainties.
117 - Do not request certain stations. When building initial candidates on stations 4 and 5, it means to keep the candidates
118 found on station 5(4) even if no compatible cluster is found on station 4(5) if that station is not requested. However,
119 if one or more compatible clusters are found we do not consider the possibility to do not attach any cluster on that
120 station. When tracking the candidates to station 3 to 1, it means to consider the possibility to continue the tracking
121 without attaching any cluster on the station that is not requested, even if compatible clusters are found on that
122 station. However the clusters found upstream after having attached cluster(s) on that station are skipped.

124 ## Examples of workflow
125 
126 - The line below allows to read the clusters from the file `clusters.in`, run the new tracking algorithm, read the
127 associated vertex from the file `vertices.in`, extrapolate the tracks to the vertex and write the result (tracks at
128 vertex, MCH tracks and associated clusters) in the file `tracks.out`:
129 
130 `o2-mch-clusters-sampler-workflow --infile "clusters.in" | o2-mch-clusters-to-tracks-workflow --l3Current 29999.998047 --dipoleCurrent 5999.966797 | o2-mch-vertex-sampler-workflow --infile "vertices.in" | o2-mch-tracks-to-tracks-at-vertex-workflow --l3Current 29999.998047 --dipoleCurrent 5999.966797 | o2-mch-tracks-sink-workflow --outfile "tracks.out"`
131 
132 - The line below allows to read the MCH tracks and associated clusters from the file `tracks.in`, refit them and write
133 them in the file `tracks.out`:
134 
135 `o2-mch-tracks-sampler-workflow --infile "tracks.in" --forTrackFitter | o2-mch-tracks-to-tracks-workflow --l3Current -30000.025391 --dipoleCurrent -5999.954590 | o2-mch-tracks-sink-workflow --outfile "tracks.out" --mchTracksOnly`
